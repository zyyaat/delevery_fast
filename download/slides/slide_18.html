SYNC vs ASYNC

Communication Patterns / أنماط التواصل

When to use gRPC vs Kafka vs WebSocket — picked per use case, not as a religion.

bolt

Synchronous · gRPC

المتزامن
Use cases
  • Order Service → Fraud Service (need risk score before confirming)
  • Order Service → Payment Service (need confirmation before notifying restaurant)
  • Customer App → API Gateway → any read query
Properties: Low latency (<100ms p99), circuit breakers via go-resiliency, retries with jitter, timeouts at 2s.
Failure mode: Caller fails fast — no cascading blocking.
stream

Asynchronous · Kafka

غير المتزامن
Use cases
  • OrderCreated → Notification, Analytics, Loyalty (fan-out)
  • DriverLocation → Geo Service, Command Center (high-volume stream)
  • MenuUpdated → Catalog, Search (eventual consistency OK)
Properties: Throughput 50K+ msgs/sec, retention 7–30 days, Avro schemas, exactly-once for payments.
Failure mode: Consumer lag — alert if >1000 msgs behind.
wifi_tethering

Real-time · WebSocket

الوقت الحقيقي
Use cases
  • Customer App — order tracking (driver location every 5s)
  • Driver App — new order pushes (sub-second)
  • Command Center — live ops dashboard
Properties: WS Gateway service, 100K concurrent connections per pod, horizontal scaling via Redis pub/sub.
Failure mode: Client auto-reconnects with exponential backoff; missed events fetched via REST.
rule
DECISION RULE: if the caller NEEDS the result to proceed → gRPC. If the caller just announces something happened → Kafka. If clients need push → WebSocket.

TECHNICAL BLUEPRINT v1.0
- Three patterns, three different problems — never default to one. - gRPC for need-it-now dependencies (Fraud, Payment, reads). - Kafka for fan-out and high-volume streams (events, location). - WebSocket for client push (tracking, dispatch, dashboards). - Decision rule at the bottom is the takeaway — print it on the wall.
